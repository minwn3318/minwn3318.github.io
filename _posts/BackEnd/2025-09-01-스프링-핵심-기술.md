# 스프링 핵심 기술
스프링은 자바 플랫폼을 위한 프레임워크로, 객체지향 특징이 강화되어있다. 아래는 스프링 프레임워크를 **사용하기 위한 핵심 기술** 설명이다

## 1. 객체지향 프로그래밍(OOP)
자바의 핵심 설계 지론인 객체지향을 지지한다. 스프링 프레임 워크는 해당 지론을 바탕으로 프로그램을 설계한다

### (1) 객체지향 설계 원칙(SOLD)
SRP : 단일 책임의 원칙
- 클래스는 단 한 개의 책임만을 가져야 한다
- 응집도는 높게, 결합도는 낮게 설계하는 것을 지향한다

**예시**
```
public class Hanul {
    private Galaxy phone;

    public Hanul(Galaxy phone) {
        this.phone = phone;
    }

    public void usePhone() {
    }
}
``` 
> 객체가 다른 객체를 생성하는 책임을 제거하고 생성자를 통해 외부에서 객체를 주입받는 모습임

 
### (2) 개방-폐쇄 원칙 (OCP)
- 확장에는 열려 있어야 하고 변경에는 닫혀 있어야 한다
- 자주 변화하는 부분을 추상화하여, 수정하지 않고도 기능을 확장시킬 수 있게 설계해, 유연성과 재사용성, 유지보수성을 높이는 것이 특징이다

**예시**
```
public class Hanul {
    private Galaxy phone;

    public Hanul(Galaxy phone) {
        this.phone = phone;
    }

    public void usePhone() {
    }
}
```
> 핸드폰이 변경되더라도 기존 코드가 유지된다

 
### (3) 리스코프 치환 원칙 (LSP)
- 부모객체를 호출하는 동작에서 자식 객체가 부모 객체를 완전히 대체할 수 있어야한다

**예시**
```
// 부모
public interface Phone {
    void powerOn();
    void useSamsungPay();
}

// 자식
public class Galaxy implements Phone {
    @Override
    public void powerOn() {
        …
    }
    @Override
    public void useSamsungPay() {
        …
    }
}

// 호출 시
public static void main(){
    Phone phone = new Galaxy()
}
```
> 자동형변환 덕분에 부모타입 변수에 자식타입 객체가 들어가도, 자식 객체를 부모타입처럼 여겨져 사용할 수 있다

 
### (4) 인터페이스 분리 원칙 (ISP)
- 객체는 자신이 호출하지 않는 메소드에 의존하지 않아야 한다

**예시**
```
// 부모
public interface Phone {
    void powerOn();
    void Pay();
}

// 자식 1
public class Galaxy implements Phone {
    @Override
    public void powerOn() {
        …
    }
    @Override
    public void Pay() {
        …
    }
}

// 자식 2
public class Iphone implements Phone {
    @Override
    public void powerOn() {
        …
    }
    @Override
    public void Pay() {
        …
    }
}
```
> 인터페이스에서 공통되는 메서드를 제공함으로써, 객체들은 다른 객체가 필요로 하는 메서드를 만들 이유가 없어졌다.
 
 
### (5) 의존관계 역전 원칙 (DIP)
- 상위 모듈이 하위 모듈을 사용할 때, 하위 모둘의 **구체적인 구현체**에 직접 의존하지 말고, **인터페이스**에 의존하라는 원칙이다

**예시**
```
List<String> list1 = new ArrayList()<>;

Map<int,String> map1 = new HashMap()<>;
```
> ArrayList나 HashMap 자료형을 인스턴스화 할 때, 변수 타입
을 List나 Map의 인터페이스 타입으로 선언하여 인스턴스화 하는 것이 그 예시다

 
## 2. 제어의 역전 (IoC)
개발자가 아닌 프레임워크가 직접 객체간의 제어를 맡음으로써 개발자들은 설계에 더 시간을 쓸 수 있게 되었다. 아래는 제어 역전 기술들이다

### (1) 컨테이너 (Container) && 빈 (Bean)
**컨테이너**
스프링이 생성한 객체인 빈을 제어하는 공간이다
   
- 컨테이너 종류
    **빈 팩토리**
    빈을 등록하고 객체 생성, 조회, 호출 등의 다양한 기능을 담당한다

    **애플리케이션 컨텍스트**
    빅 팩토리 확장판으로 초기화 시점에 빈을 생성해 두어 필요할 때 즉시 빈을 사용할 수 있다

**빈**
스프링이 제어하는 객체다
      
- 빈 등록 어노테이션
    1. @Bean, @Configuration
        빈 객체, 수동으로 등록할 때 사용된다
        **예시**
        ```
        @Configuration
        public class AppConfig {
            @Bean
            public Phone phone() {
                return new Galaxy();
            }
            @Bean
            public Hanul hanul {
                return new Hanul();
            }
        }
        ```

    2. @Controller
        사용자로부터 요청이 들어오면 해당 업무를 처리하는 모델을 호출하고 결과를 뷰에 전달하는 객체
        - 보통 RESTful 컨트롤러층에 사용된다

        **예시**
        ```
        @Controller
        public class MyController {
            @GetMapping("/greeting")
            public String greeting() {
                return "Hello, World!";
            }
        }
        ```

    3. @Repository
        사용자의 요청을 받고. 데이터베이스와 통신할 때 사용되는 호출부 객체
        - 보통 dto와 db를 연결하는 객체 Repository층에 사용된다

        **예시**
        ```
        @Repository
        public class MyRepository {
            public String getData() {
                return "Data from database";
            }
        }
        ```
    
    4. @ComponentScaen
        컴포넌트 스캔 객체
        - 메인 클래스에서 사용된다
        **예시**
        ```
        @Service
        public class MyService {
            public String provideService() {
                return "Service has been provided";
            }
        }
        ```
    5. @Component
        컴포넌트, 스프링이 스스로 클래스를 빈으로 등록할 때 사용된다
        - 객체로 만들고자 하는 클래스에서 사용된다
        - 서비스층에 보통 쓰이지만, @Service가 더 쓰이는 추세
        **예시**
        ```
        @Component
        public class MessageService {

            public String getMessage() {
                return "Hello World";
            }
        }
        ```
   
## 3. 의존성 주입 (DI)
**의존성이란**, 간단히 말해 한 객체가 다른 객체를 참조하기 위해 생성한 관계를 가리킨다. 이 의존성이 높을 수록 결합도가 강해, 확장성이 낮아진다. 스프링은 이러한 점을 해결하기 위해 의존성 낮추는 방식으로, 인터페이스와 같은 구성 요소에 의존하도록 설계하였다

### (1) 의존성 주입 (Dependency Injection)

1. 생성자 주입
    생성자를 통해서 의존관계를 주입하는 방법이다
    @Autowired를 사용한다
    - 1번만 호출되는 것을 보장한다
    - 불변과 필수 의존관계에 사용된다

    **예시**
    ```
    @Component
    public class Hanul {
    private final Phone phone;
        @Autowired
        public Hanul(Phone phone) {
            this.phone = phone;
        }
    }
    ```
2. 수정자 주입
    setter메서드에 @Autowired를 붙여 사용된다
    - 중간에 의존관계를 변경시킬 수 있다
    **예시**
    ```
    @Component
    public class Hanul {
    private final Phone phone;
        @Autowired
        public setPhone(Phone phone) {
            this.phone = phone;
        }
    }
    ```

3. 필드 주입
    필드변수 자체에 @Autowired를 붙여 사용된다
    - 매우 직관적이다
    - 그러나 순수 자바코드에서는 작동되지 않는다.
    **예시**
    ```
    @Component
    public class Hanul {
    private final Phone phone;
        @Autowired
        private Phone phone;   
    }
    ```
